# 품질 검토 기준

Phase 4에서 생성된 docs/ 전체를 검증하는 기준을 정의한다.

---

## 1. 문서 간 정합성

문서들 사이에 모순이 없는지 교차 검증한다.

### 1.1 데이터 정합성

- product-spec에서 정의한 API의 request/response 타입이 `data-model.md`의 엔티티 필드와 일치하는가
  - 예: product-spec에서 `Todo.priority`를 `number`로 정의했는데 data-model에서 `string`이면 **불일치**
- product-spec의 에러 코드가 auth.md나 다른 설계 문서와 일치하는가
- 엔티티 간 관계가 모든 문서에서 동일하게 기술되었는가

### 1.2 구조 정합성

- `ARCHITECTURE.md`의 디렉토리 구조가 `AGENTS.md`의 Directory Map과 일치하는가
  - 한쪽에만 있는 경로가 없어야 한다
- design-docs/ 간에 모순되는 정의가 없는가
  - 예: core-beliefs.md에서 "전역 상태 라이브러리 사용 금지"라고 했는데, 어떤 product-spec에서 Redux를 사용한다고 되어 있으면 **모순**

### 1.3 참조 정합성

- `index.md` 파일에 나열된 문서가 실제로 존재하는가
- product-spec의 의존성 섹션에서 참조하는 다른 기능/모듈이 실제 product-spec으로 존재하는가

**판정**: 불일치가 1건이라도 있으면 **실패**. 수정 후 재검증.

## 2. 표현 품질

문서의 표현이 구체적이고 측정 가능한지 확인한다.

### 2.1 금지 표현

다음 표현이 단독으로 사용되면 위반:

| 금지 표현 | 위반 예시 | 대체 예시 |
|----------|---------|----------|
| "적절한" | "적절한 에러 처리" | "네트워크 에러 시 3초 타임아웃 후 토스트 알림" |
| "필요시" | "필요시 캐싱 도입" | "목록이 100개 초과 시 React.memo로 렌더링 최적화" |
| "충분한" | "충분한 성능" | "초기 로딩 1초 이내, LCP 2.5초 이내" |
| "등등" / "기타" | "인증, 권한 등등 처리" | "인증(Firebase Auth), 권한(Firestore Security Rules) 처리" |
| "TBD" | "배포 방식: TBD" | 해당 항목을 채우거나, 명시적으로 "v1 범위 밖. Phase 2에서 결정 예정" |
| "추후 작성" | "접근성: 추후 작성" | 해당 항목을 채우거나, 명시적으로 "v1에서는 기본 키보드 네비게이션만. 상세는 v2에서" |
| "TODO" | "TODO: API 설계" | 해당 항목을 채우거나, gap-report에 미충족으로 기록 |

**예외**: 문맥 안에서 구체적 설명과 함께 사용되면 허용. "필요시 캐싱을 도입한다 — 구체적으로 목록 100개 초과 시 검토" → OK.

### 2.2 측정 가능성

| 영역 | 기대하는 수치 예시 |

## 4. 점수 산출

### 4.1 product-spec 점수

각 product-spec의 체크리스트 항목 점수를 평균낸다.

```
페이지별 점수 = sum(항목 점수) / count(항목, N/A 제외)
전체 product-spec 점수 = sum(페이지별 점수) / count(페이지)
```

### 4.2 아키텍처 점수

아키텍처 체크리스트 4개 항목의 평균.

```
아키텍처 점수 = sum(항목 점수) / count(항목, N/A 제외)
```

### 4.3 정합성 점수

통과 / 실패 (이진).

- 문서 간 정합성 검사에서 불일치가 0건이면 통과
- 1건이라도 있으면 실패 → 수정 후 재검사

### 4.4 표현 품질 점수

금지 표현 위반 건수.

- 0건: 우수
- 1~3건: 양호 (수정 권장)
- 4건 이상: 미흡 (수정 필수)

### 4.5 종합 등급

| 등급 | 조건 |
|------|------|
| **A (우수)** | product-spec 평균 ≥ 2.5, 아키텍처 ≥ 2.5, 정합성 통과, 표현 위반 0건 |
| **B (양호)** | product-spec 평균 ≥ 2.0, 아키텍처 ≥ 2.0, 정합성 통과, 표현 위반 ≤ 3건 |
| **C (보통)** | product-spec 평균 ≥ 1.5 또는 정합성 실패 |
| **D (미흡)** | 위 조건 모두 해당 안 됨 |
