# docs-orchestrator 설계서 평가

> 평가일: 2026-04-01
> 대상: docs-orchestrator-design.md (최종 수정 2026-04-01)

---

## 총평

잘 만든 설계서다. 문제 정의가 실제 경험에서 나왔고, 해결책이 기존 코드베이스의 구조를 존중하면서도 핵심 병목(컨텍스트 압축)을 정확히 공략한다. 22개의 설계 결정이 모두 근거와 함께 기록되어 있어, 구현 단계에서 "왜 이렇게 했지?"라는 질문이 발생하지 않을 것이다.

다만 몇 가지 실질적인 리스크와 빠진 부분이 있다.

---

## 강점

### 1. 핵심 원칙이 실전에서 검증된 문제를 정확히 해결한다

"추출, 요약 금지"와 "한 세션, 한 집중"은 LLM 오케스트레이션에서 가장 흔하게 실패하는 두 가지 패턴을 정면으로 차단한다. 특히 `<!-- SOURCE: 원본파일명#섹션 -->` 태그로 중복을 추적하면서도 self-contained를 유지하는 접근은 실용적이다.

### 2. 오케스트레이터를 파이썬 스크립트(AI 아닌)로 구현한 결정

이것이 이 설계의 가장 중요한 결정이다. 오케스트레이터 자체가 AI 세션이면 재귀적으로 같은 컨텍스트 문제에 빠진다. 기존 `session_manager.py`의 패턴(상태 머신 + 세션 열기 + idle 감지)을 재활용하면서 단위만 Chunk → Phase/도메인/기능으로 바꾸는 접근은 코드 재사용 면에서도 효율적이다.

### 3. Phase 1.5 품질 게이트

Phase 1 산출물이 불량하면 이후 전체가 무너지는데, 이걸 AI 판단이 아닌 기계적 검증(줄 수 비교, SOURCE 태그 매핑, 빈 파일 검사)으로 차단하는 설계가 좋다. `GateResult` dataclass도 기존 프로젝트의 "dataclass for data, no classes with methods" 컨벤션에 부합한다.

### 4. auto/manual/하이브리드 모드 분리

결제나 인증처럼 민감한 도메인만 manual로 지정하고 나머지는 auto로 돌리는 하이브리드 모드는 실무에서 매우 유용할 것이다. golden-rules.md의 ESCALATE 철학과도 일관된다.

### 5. 에러 복구 정책이 명확하다

§8.3의 3단계 복구(마커 확인 → 불완전 파일 삭제 → pending 되돌림)는 단순하면서도 안전하다. "불완전한 파일이 다음 세션에 전파되지 않도록 깨끗하게 시작한다"는 원칙이 특히 중요하다.

## 리스크 및 개선 필요 사항

### 리스크 1: 세션 수가 너무 많다 — 실행 시간 + 비용

대규모 프로젝트에서 예상 세션 수가 20~30개다. 각 세션은 최소 세션 열기(3초) + 프롬프트 입력(1초) + AI 작업(수 분) + idle 대기(120초) + 완료 판정이 필요하다. 이를 합산하면:

- **낙관적 추정**: 세션당 평균 5분 × 25세션 = 약 2시간
- **현실적 추정**: 세션당 평균 8~10분(문서 생성은 코드보다 느릴 수 있음) × 25세션 = 약 3~4시간
- 재시도가 발생하면 더 길어진다

이것 자체는 문제가 아닐 수 있지만(사람이 안 하니까), **비용**이 문제다. 25~30개 세션의 Claude API 호출 비용을 사용자가 예상할 수 있도록 Phase 0에서 예상 세션 수 + 예상 소요 시간을 알려주는 게 좋겠다. `orchestrator-state.json`의 `estimated_sessions`에 이미 필드가 있으니, 이걸 Phase 0에서 터미널에 출력하면 된다.

**권장 사항**: Phase 0에서 "예상 세션: N개, 예상 소요 시간: ~M분" 출력 + 사용자 확인(y/n) 추가.

### 리스크 2: idle_timeout_seconds 120초의 근거가 약하다

"문서 생성은 생각 시간이 길다"는 맞지만, 120초가 적절한지에 대한 실측 데이터가 없다. 실제 Cowork 세션에서 Read → 긴 사고 → Write 패턴의 사고 시간이 30초인지 90초인지 180초인지 모른다. 너무 짧으면 false-positive idle 판정이 나고, 너무 길면 실패한 세션을 오래 방치한다.

**권장 사항**:
- 초기값 120초는 괜찮지만, "실험 후 조정 예정"이라고 명시하는 게 좋다 (feature_bundle_threshold_lines처럼)
- 또는 Phase 1 세션을 먼저 돌려보고 실측한 후 나머지 Phase에 반영하는 "점진적 튜닝" 전략을 문서에 추가

### 리스크 3: Phase 3-B 묶음 세션에서 "RE-READ 루프"의 실효성

Phase 3 프롬프트에 "이미 읽었으니 기억한다는 생각을 하지 말라. 매 문서마다 반드시 RE-READ하라"고 지시한다. 의도는 좋지만 실제로 LLM이 이 지시를 얼마나 따르는지는 불확실하다. 특히 한 세션에서 2~3개 기능의 product-spec을 연속 생성할 때, 세 번째 문서를 쓸 때쯤이면 RE-READ 지시를 무시할 가능성이 있다.

**권장 사항**:
- 기능별 1세션을 기본으로 하고, 묶음은 "최적화"로 남겨두는 게 안전
- 또는 Phase 4-2 체크리스트 재평가에서 묶음 세션의 후반 문서에 품질 저하가 있는지 체크하는 항목 추가

### 리스크 4: `<!-- ORCHESTRATOR:DONE -->` 마커 신뢰성

완료 마커는 AI 세션이 프롬프트 지시를 따라 "마지막 줄에 기록"하는 것에 의존한다. AI가 마커를 빼먹으면 오케스트레이터는 세션이 끝났는데도 "불완전"으로 판정한다. 특히 세션이 길어져서 컨텍스트 압축이 발생하면(이 설계가 해결하려는 바로 그 문제), 프롬프트 후반의 마커 지시를 잊을 수 있다.

**권장 사항**:
- 보조 검증 추가: 마커가 없지만 파일 내용이 충분하면(예: 최소 줄 수 충족 + 금지 표현 미포함), "마커 누락 의심 — 내용 기반으로 완료 판정" 같은 fallback 로직
- 또는 마커를 프롬프트 첫 줄과 마지막 줄 양쪽에 지시하여 한쪽이라도 살아남게

### 리스크 5: Phase 4-1 정합성 검사의 범위 제한

"§4 데이터 & API 섹션만" 발췌해서 비교한다고 되어 있다. 하지만 product-spec의 섹션 구조가 프로젝트 타입에 따라 다르다면(web-app은 11섹션, api는 8~10섹션 등), "§4"가 항상 "데이터 & API"인지 보장되지 않는다. 실제 output-formats.md의 템플릿에 의존하는 부분인데, 이 참조 파일의 내용이 설계서에 포함되어 있지 않아 검증이 어렵다.

**권장 사항**: 섹션 번호가 아니라 섹션 제목(예: "데이터 & API"라는 헤더)으로 매칭하도록 명시

아래 5개 항목은 설계서 §12에 구현 상세로 반영 완료:

5. **`_summary.md` 생성 주체** → §12.5: 오케스트레이터(파이썬)가 정규식으로 기계적 합산, `generate_gap_summary()` 함수

## 기존 코드베이스와의 정합성

| 항목 | 판단 | 비고 |
|------|------|------|
| `session_opener.open_new_session()` 재활용 | **호환** | 함수 시그니처(`initial_prompt`) 그대로 사용 가능 |
| `session_finder.find_active_jsonl()` 재활용 | **호환** | JSONL 감지 로직 그대로 |
| `is_idle_trigger()` 재활용 | **주의** | `idle_timeout_seconds` 파라미터를 받으므로 120초 전달 가능. 단, 현재 harness의 `run_harness()`에서 `harness_config.idle_timeout_seconds`를 직접 넘기므로 docs-orchestrator도 같은 패턴으로 처리해야 함 |
| `WatcherStateMachine` 재활용 | **호환** | auto 모드에서 cooperative loop 동일 패턴 |
| `Config` dataclass 확장 | **필요** | `DocsOrchestratorConfig` 추가 + `load_docs_orchestrator_config()` 함수 필요 |
| AGENTS.md 컨벤션 | **호환** | "Each file has one responsibility" 준수 |
| main.py CLI 확장 | **필요** | `--mode docs-orchestrator` 추가 + `--docs-mode` 추가 |
| `models.py` | **확장 불필요** | 기존 Event/Response 모델은 Phase 1 Watch에서만 사용. 오케스트레이터 상태는 JSON으로 별도 관리 |
