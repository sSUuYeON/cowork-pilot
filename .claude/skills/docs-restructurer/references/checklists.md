# 갭 분석 체크리스트

Phase 2에서 사용하는 체크리스트. 입력 문서의 각 항목에 점수를 매겨 정보 충분도를 평가한다.

---

## 점수 체계

| 점수 | 의미 | 판단 기준 |
|------|------|----------|
| **3 (충족)** | 구체적이고 측정 가능하게 정의됨 | 개발자가 이 문서만 보고 구현할 수 있는 수준 |
| **2 (부분 충족)** | 정보는 있으나 모호하거나 불완전 | 방향은 알겠으나 세부사항이 빠져 구현 시 판단이 필요 |
| **1 (미충족)** | 해당 정보 없음 | 언급 자체가 없거나 "미정"/"TBD"로만 표기 |
| **N/A (해당 없음)** | 프로젝트 특성상 불필요 | 예: CLI 앱에 UI 상태 체크리스트는 N/A |

## product-spec 체크리스트 (모든 프로젝트 타입)

각 페이지/기능별로 아래 8개 항목을 평가한다.

| # | 항목 | 기준 | 충족 (3점) 예시 | 미충족 (1점) 예시 |
|---|------|------|----------------|-----------------|
| 1 | **문제 정의** | 이 기능이 해결하는 문제가 구체적으로 서술되었는가 | "사용자가 할 일의 마감일을 놓치는 문제. 현재 30%의 할 일이 마감일을 초과한다" | "할 일 관리 기능" (문제가 아닌 기능 설명) |
| 2 | **유저 플로우** | 모든 유저 액션에 대한 시스템 반응이 step-by-step으로 정의되었는가 | "1. 사용자가 + 버튼 클릭 → 인라인 폼 표시 2. 제목 입력 후 Enter → Firestore에 저장, 목록에 즉시 추가 3. ..." | "할 일을 추가할 수 있다" (step이 없음) |
| 3 | **에러 패스** | 해피 패스 외에 에러 시나리오가 유형별로 정의되었는가 | "네트워크 끊김 시 로컬 캐시에 저장, 재연결 시 동기화. 제목 빈 문자열 시 빨간 테두리 + '제목을 입력하세요'" | 에러 시나리오 언급 없음 |
| 4 | **데이터 스키마** | API/함수의 request/response 타입이 정의되었는가 | `interface CreateTodoRequest { title: string; categoryId?: string; priority: Priority; dueDate?: Date }` | "할 일 데이터를 저장한다" |
| 5 | **에러 처리** | 에러 조건, 사용자 메시지, 복구 동작이 테이블로 정의되었는가 | 에러 테이블: `| auth/wrong-password | 비밀번호가 틀렸습니다 | 입력 필드 포커스 + 재시도 |` | "에러 시 적절히 처리" |
| 6 | **성능 기준** | 측정 가능한 수치가 최소 1개 이상 명시되었는가 | "목록 로딩 200ms 이내, 한 번에 50개까지 표시, 이후 무한 스크롤" | "빠르게 로딩되어야 한다" |
| 7 | **의존성** | 다른 기능/모듈과의 의존 관계가 명시되었는가 | "AuthProvider에 의존 (user.uid 필요). Category 모듈의 카테고리 목록을 참조" | 의존성 언급 없음 |
| 8 | **Non-Goals** | 명시적으로 하지 않을 것이 정의되었는가 | "v1에서 하지 않는 것: 할 일 공유, 팀 기능, 오프라인 모드" | Non-Goals 섹션 없음 |

|---|------|------|----------------|-----------------|
| 9 | **UI 상태** | 로딩/빈/에러 상태의 화면이 각각 정의되었는가 | "로딩: 스켈레톤 UI 3행. 빈 상태: '아직 할 일이 없습니다' + 추가 버튼. 에러: 토스트 알림 + 재시도" | UI 상태 언급 없음 |

프로젝트 타입이 `api` 또는 `library`이면 위 8개에 추가로 아래 2개를 적용한다 (9~11 대신).

| # | 항목 | 기준 | 충족 (3점) 예시 | 미충족 (1점) 예시 |
|---|------|------|----------------|-----------------|
| 9 | **엔드포인트 상세** | HTTP 메서드, 경로, 인증, rate limit, request/response 스키마가 있는가 | `POST /api/todos - Auth: Bearer token - Rate: 100/min - Body: CreateTodoRequest - Response: Todo` | "REST API를 제공한다" |

| 1 | **기술 선택 근거** | 각 기술에 대한 선택 이유와 버린 대안이 있는가 | "Next.js 선택: SSR + 파일 기반 라우팅. 대안 Vite+React 검토했으나 SEO 필요로 배제" | "Next.js 사용" |
| 2 | **디렉토리 구조** | 실제 파일 경로로 구조가 명시되었는가 | `src/app/` 라우트, `src/lib/` 비즈니스 로직, `src/components/` UI, `src/types/` 타입 | "src 폴더에 코드를 넣는다" |

## 미충족/부분 충족 시 행동

### 미충족 (1점)

사용자에게 AskUserQuestion으로 질문한다. 반드시 구체적 선택지를 포함한다.

예시:
> "대시보드 페이지의 '에러 처리'가 정의되지 않았습니다. 어떻게 할까요?"
> 1. 네트워크 에러 시 토스트 알림 + 재시도 버튼
> 2. 에러 시 에러 바운더리 페이지로 대체
> 3. 다른 방식 (직접 설명)

### 부분 충족 (2점)

구체화 방향을 제안하고 사용자 확인을 받는다.

예시:
> "대시보드 페이지의 '성능 기준'이 '빠르게 로딩'으로만 되어 있습니다. 이렇게 구체화하면 될까요?"
> → "초기 로딩 1초 이내, 할 일 목록 최대 50개 표시, 이후 무한 스크롤"

### 모든 항목 충족 (3점) 또는 N/A

별도 질문 없이 다음 Phase로 자동 진행한다.

## 중단 기준

미충족 항목(1점)이 **전체 항목의 50% 이상**이면 경고를 표시한다:

> "입력 문서에 정보가 너무 부족합니다."

사용자에게 선택지를 제시한다:
1. **Cowork에서 더 상의한 후 다시 시도** — 스킬을 중단하고, 사용자가 더 많은 정보를 준비한 후 다시 실행
2. **부족한 항목을 지금 여기서 하나씩 채우기** — AskUserQuestion으로 모든 미충족 항목을 순서대로 질문
3. **부족한 채로 진행** — 품질 저하를 감수하고 Phase 3으로 진행. QUALITY_SCORE.md에 경고 표시
