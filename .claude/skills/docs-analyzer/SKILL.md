# docs-analyzer

대규모 프로젝트의 자유 형식 문서들을 분석하고, 도메인별로 분할하여, `docs-generator`가 새 세션에서 고품질 문서를 생성할 수 있도록 준비한다.

## 이 스킬이 존재하는 이유

기존 `docs-restructurer`는 분석부터 생성까지 한 세션에서 모두 처리한다. 중소규모 프로젝트에서는 잘 동작하지만, 대규모 프로젝트(product-spec 5개 이상)에서는 컨텍스트가 길어지면서 품질이 떨어진다. 이 문제를 해결하기 위해 **분석(Phase 1~2)과 생성(Phase 3~5)을 별도 세션으로 분리**한다.

이 스킬은 분석만 담당한다. 생성은 `docs-generator` 스킬이 새 세션에서 수행한다.

## 사용 시점

- 예상되는 product-spec이 **5개 이상**인 프로젝트
- 입력 문서가 **10개 이상**이거나 전체 분량이 매우 큰 경우
- 기존 `docs-restructurer`로 시도했으나 품질 문제가 발생한 경우

중소규모 프로젝트(product-spec 4개 이하)에는 기존 `docs-restructurer`를 사용한다.

## 전제 조건

1. **프로젝트 폴더 접근**: 대상 프로젝트 폴더가 마운트되어 있어야 한다
2. **입력 문서**: 자유 형식 마크다운 파일 1개 이상
3. **project-conventions.md**: 있으면 그 규격을 따르고, 없으면 `references/output-formats.md` 기본 규격 사용

## Phase 1: 입력 읽기 & 구조 분석

**목표**: 입력 문서에서 어떤 정보가 있고, 무엇이 빠졌는지 파악한다. 도메인별 분할 계획을 수립한다.

**실행 단계:**

1. 마운트된 폴더를 탐색하여 마크다운 파일을 모두 수집한다. 세션 내 파일도 포함.

2. 각 파일의 내용을 읽고, 다음 입력 매핑 테이블에 따라 분류한다:

| 입력에서 발견되는 정보 | 매핑되는 docs/ 파일 |
|----------------------|-------------------|
| 프로젝트 개요, 기술 스택, 설계 철학 | `AGENTS.md`, `ARCHITECTURE.md` |
| 데이터 모델 (엔티티, 관계, 스키마) | `docs/design-docs/data-model.md` |
| 운영 원칙, 기술 철학 | `docs/design-docs/core-beliefs.md` |
| 인증/보안 설계 | `docs/design-docs/auth.md`, `docs/SECURITY.md` |
| 배포 전략 | `docs/design-docs/deployment.md` |
| 페이지/기능별 상세 스펙 | `docs/product-specs/{페이지}.md` |
| 디자인 가이드라인 | `docs/DESIGN_GUIDE.md` |
| 비기능 요구사항 (성능, 접근성 등) | 해당 문서에 분산 배치 |

3. 프로젝트 타입을 추론한다: `web-app` / `api` / `cli` / `library` / `mobile`. 추론할 수 없으면 사용자에게 직접 질문한다.

4. **도메인 분할 계획 수립**: 생성할 문서 목록을 확정하고, 도메인별 그룹으로 분류한다. 도메인 간 의존성도 파악한다.

5. 결과를 `docs/generated/analysis-report.md`에 저장한다. **기존 docs-restructurer보다 풍부하게 작성한다** — Phase 3~5가 별도 세션에서 실행되므로, 이 파일이 유일한 맥락 전달 수단이다:

```markdown
# 입력 분석 보고서

> 생성일: {YYYY-MM-DD}
> 프로젝트 타입: {추론된 타입}
> 규모: 대규모 (예상 문서 수: {N}개)

## 프로젝트 개요

{프로젝트가 해결하는 문제, 대상 사용자, 핵심 가치 — 2~3문장}

## 핵심 설계 결정

{사용자가 내린 주요 기술적/설계적 판단과 그 이유. 새 세션의 Opus가 "왜 이런 구조인지"를 이해할 수 있도록}

| 결정 | 선택 | 이유 | 검토한 대안 |
|------|------|------|-----------|
| {결정} | {선택} | {이유} | {대안과 배제 이유} |

## 입력 파일 목록

| 파일 | 주요 내용 요약 | 관련 도메인 |
|------|--------------|-----------|
| {파일명} | {1줄 요약} | {도메인 A, B, ...} |

## 정보 매핑 계획

| 카테고리 | 출처 파일 | 대상 docs/ 파일 | 정보 충분도 |
|---------|---------|----------------|------------|
| {카테고리} | {출처} | {대상} | 충분 / 부분 / 없음 |

## 도메인 분할 계획

### 생성 그룹

| 그룹 | 문서 | 의존성 | 실행 순서 |
|------|------|--------|---------|
| A: 설계 기반 | core-beliefs.md, data-model.md, ... | 없음 | 1 (먼저) |
| B-1: {도메인1} | {페이지 목록} | 그룹 A | 2 |
| B-2: {도메인2} | {페이지 목록} | 그룹 A, B-1 | 3 (B-1에 의존) |
| B-3: {도메인3} | {페이지 목록} | 그룹 A | 2 (B-1과 병렬 가능) |
| C: 아키텍처 | ARCHITECTURE.md, ... | 그룹 A, B 전체 | 4 |
| D: 진입점 | AGENTS.md | 전체 | 5 (마지막) |

### 도메인 간 의존성

{도메인 간 참조 관계를 구체적으로 기술. 예: "예약 도메인의 product-spec이 결제 도메인의 Payment 엔티티를 참조한다"}

### 병렬 실행 가능 그룹

- 병렬 가능: B-1, B-3 (상호 의존 없음)
- 순차 필요: B-1 → B-2 (B-2가 B-1의 데이터 모델 참조)

## 원본 파일 → 도메인 매핑

| 원본 파일 | 해당 도메인 | Phase 3에서 참조 필요 여부 |
|----------|-----------|----------------------|
| {원본 파일} | {도메인} | 예/아니오 |

## 누락 항목

- {발견되지 않은 정보 목록}
```

6. **도메인별 추출 파일 생성**: 원본 기획서에서 도메인별로 관련 내용을 추출하여 `docs/generated/domain-extracts/`에 저장한다:

```
docs/generated/domain-extracts/
├── shared.md         — 모든 도메인이 공유하는 정보 (기술 스택, 설계 원칙 등)
├── {도메인1}.md      — 도메인1 관련 원본 내용 추출
├── {도메인2}.md      — 도메인2 관련 원본 내용 추출
└── ...
```

각 도메인 추출 파일에는 해당 도메인의 product-spec을 작성하는 데 필요한 원본 내용을 담는다. Phase 3 세션이 원본 전체를 읽지 않고 해당 도메인 추출 파일만 읽으면 충분하도록, 필요한 맥락을 빠뜨리지 않는다.

7. 사용자에게 분석 결과를 요약 보고한다. 누락 항목과 도메인 분할 계획을 함께 보여준다.

## Phase 2: 갭 분석 + 점수 평가

**사전 작업:**
- `references/checklists.md`를 읽는다
- `docs/generated/analysis-report.md`를 다시 읽는다

**실행 단계:**

1. **product-spec 체크리스트 적용**: 각 페이지/기능별로 `references/checklists.md`의 "product-spec 체크리스트"를 적용한다. 각 항목에 점수(3/2/1/N/A)를 매긴다.

2. **프로젝트 타입별 추가 체크리스트 적용**: 프로젝트 타입이 `web-app`이면 "web-app 추가 체크리스트"도 적용. `api`/`library`이면 해당 체크리스트 적용.

3. **아키텍처 체크리스트 적용**: 프로젝트 전체에 대해 아키텍처 체크리스트를 적용한다.

4. **미충족/부분 충족 항목 처리**:
   - **미충족 (1점)**: 사용자에게 AskUserQuestion으로 질문한다. 구체적 선택지를 제시.
   - **부분 충족 (2점)**: 구체화 방향을 제안하고 확인.
   - **충족 (3점) 또는 N/A**: 별도 조치 없이 통과.

5. **중단 기준**: 미충족 항목이 전체의 50% 이상이면 경고:
   > "입력 문서에 정보가 너무 부족합니다. (1) Cowork에서 더 상의한 후 다시 시도 (2) 부족한 항목을 지금 여기서 채우기 (3) 부족한 채로 진행 (품질 저하 감수)"

6. 결과를 `docs/generated/gap-report.md`에 저장한다. **사용자 응답을 풍부하게 기록한다** — 설계 의도, 우선순위 판단, 방향성("이건 심플하게", "v2에서 하자" 등)도 포함:

```markdown
# 갭 분석 보고서

> 생성일: {YYYY-MM-DD}

## 종합 점수: {평균} / 3.0

## 페이지별 product-spec 점수

| 페이지 | 문제정의 | 유저플로우 | 에러패스 | 데이터 | 에러처리 | 성능 | 의존성 | Non-Goals | 평균 |
|--------|---------|----------|---------|-------|---------|------|-------|----------|------|
| {페이지} | {점수} | ... | | | | | | | {평균} |

## 아키텍처 점수

| 항목 | 점수 | 비고 |
|------|------|------|
| {항목} | {점수} | {설명} |

## 보강된 내용

{AskUserQuestion으로 받은 사용자 응답을 여기에 기록. 단순 답변뿐 아니라 사용자의 설계 의도와 방향성도 함께 기록한다}

### {페이지/항목명}

- **질문**: {무엇을 물었는지}
- **사용자 응답**: {답변 내용}
- **설계 의도**: {왜 이런 선택을 했는지, 사용자가 말한 맥락}

## 사용자 설계 방향성

{Phase 2 과정에서 파악된 사용자의 전반적인 설계 방향. 예: "MVP 우선, 최소한의 기능으로 빠르게", "보안은 꼼꼼하게, 성능은 나중에 최적화"}
```

7. 사용자에게 점수 요약을 보고한다.

## Phase 2 완료 후: 다음 세션 안내

Phase 2가 완료되면, 사용자에게 다음 세션에서 `docs-generator` 스킬로 Phase 3~5를 실행하는 방법을 안내한다.

**저장된 파일 검증**: 먼저 다음 파일들이 모두 정상 저장되었는지 확인한다:

- `docs/generated/analysis-report.md`
- `docs/generated/gap-report.md`
- `docs/generated/domain-extracts/shared.md`
- `docs/generated/domain-extracts/{도메인별}.md`

**다음 세션 프롬프트 템플릿**: 도메인 분할 계획에 따라 실행 순서를 안내한다:

> "분석이 완료되었습니다. 이제 새 세션에서 `docs-generator` 스킬로 문서를 생성하면 됩니다.
>
> **실행 순서:**
> 1. 그룹 A (설계 기반): `docs-generator`를 실행하며 도메인 범위를 '그룹 A: design-docs'로 지정
> 2. 그룹 B (product-specs): 도메인별로 세션을 나눠서 실행
>    - {도메인1}과 {도메인3}은 병렬 실행 가능
>    - {도메인2}는 {도메인1} 완료 후 실행 (의존성)
> 3. 그룹 C (아키텍처): 그룹 B 전체 완료 후
> 4. 그룹 D (AGENTS.md): 마지막
>
> 각 세션에서 이렇게 프롬프트하면 됩니다:
> '`docs-generator` 스킬을 사용해서 {프로젝트 경로}의 {그룹/도메인} 문서를 생성해줘'"

## 에러 처리

### 입력 문서가 너무 얕은 경우
Phase 2 갭 분석에서 미충족 항목 50% 이상이면 경고 표시.

### 프로젝트 타입 판별 실패
사용자에게 직접 질문.

### 파일 쓰기 실패
폴더 마운트 요청.
