# Decision Criteria

> CLI 에이전트가 Cowork 질문에 응답할 때 참고하는 판단 기준.
> 이 문서는 실제 Cowork 세션 데이터 분석 (4세션, ~450 tool_use, 11 AskUserQuestion) 기반으로 작성됨.

---

## 1. 도구 승인 (Tool Approval) — 전체의 96%

도구 승인이 가장 빈번한 이벤트다. 도구별로 판단 기준이 다르다.

### 무조건 허용 (Auto-Allow)

이 도구들은 읽기 전용이거나 부작용이 없다. 즉시 `allow`:

| 도구 | 이유 |
|------|------|
| `Read` | 파일 읽기만. 부작용 없음. 실제 데이터에서도 170건 전부 안전. |
| `Glob` | 파일 탐색만. 부작용 없음. |
| `Grep` | 텍스트 검색만. 부작용 없음. |
| `TodoWrite` | 진행 추적. 외부 영향 없음. |
| `WebSearch` | 검색만. 부작용 없음. |

### 조건부 허용 (Conditional Allow)

프로젝트 경로 내에서만 허용. golden-rules 블랙리스트 체크 필요:

| 도구 | 허용 조건 | 거부 조건 |
|------|-----------|-----------|
| `Write` | 프로젝트 디렉토리 내 파일 생성 | 시스템 경로(`/etc`, `/usr`, `~/.ssh`), `.env`에 실제 시크릿 값 |
| `Edit` | 프로젝트 디렉토리 내 파일 수정 | 시스템 파일, 보안 관련 파일 |
| `Bash` | 아래 '안전한 명령' 목록에 해당 | 아래 '위험한 명령' 목록에 해당 |
| `Agent` | 서브에이전트 분석/작업 | 프롬프트에 외부 서비스 접근 지시 포함 |
| `WebFetch` | 공개 URL 접근 | 인증 필요 URL, 내부 서비스 URL |

### Bash 명령 세부 기준

**안전한 명령 (allow):**
```
ls, find, cat, head, tail, wc          # 읽기 전용
mkdir, touch                            # 디렉토리/파일 생성
git status, git log, git diff, git add  # git 읽기 + 스테이징
git commit                              # 커밋 (되돌릴 수 있음)
npm install, npm run build, npm test    # 패키지 관리 + 빌드
pip install, pytest                     # Python 패키지 + 테스트
go build, go test, go mod tidy          # Go 빌드 + 테스트
echo, printf                            # 출력만
cp, mv (프로젝트 내)                     # 프로젝트 내 이동/복사
```

**위험한 명령 (deny → golden-rules 참조):**
```
rm -rf, rm -r                           # 재귀 삭제
git push --force                        # 강제 푸시
git reset --hard                        # 이력 파괴
chmod, chown                            # 권한 변경
curl/wget + POST (외부 API)              # 외부 데이터 전송
sudo 어떤것이든                          # 루트 권한
kill, pkill                             # 프로세스 종료
docker rm, docker rmi                   # 컨테이너/이미지 삭제
ssh, scp                                # 원격 접근
```

**판단 필요 (CLI 에이전트가 맥락 보고 결정):**
```
git push (일반)                          # 원격 반영 — 프로젝트 문서에서 허용 여부 확인
npm publish                              # 패키지 배포 — 보통 deny
rm (단일 파일, 프로젝트 내)               # 맥락에 따라 allow 가능
```

### 무조건 거부 (Auto-Deny)

golden-rules.md 블랙리스트에 해당하면 무조건 `deny`. → golden-rules.md 참조.

## 2. AskUserQuestion (질문 응답) — 전체의 2.5%

빈도는 낮지만 판단 복잡도가 높다. 카테고리별 전략:

### 2.1 아키텍처/설계 결정

**실제 예시:**
- "이 3단계 분할로 진행할까요?"
- "Service/Repo 레이어를 미리 골격을 잡아둘까요, 최소한만?"
- "SSE 실시간 업데이트를 v1에 포함할까요?"

**판단 기준:**
1. **프로젝트 스펙/기획서에 명시된 내용이 있으면** → 그에 맞는 옵션 선택
2. **(권장) 표시가 있으면** → 높은 가중치로 고려 (Cowork가 분석 후 권장한 것)
3. **YAGNI/KISS 원칙** → 옵션 중 "최소한", "간단한", "나중에" 쪽이 보통 안전
4. **확신 없으면** → "(권장)" 표시된 옵션 선택
5. **프로젝트 문서와 완전히 충돌하면** → ESCALATE

### 2.2 UX/UI 결정

**실제 예시:**
- "로그인 후 어디로 보낼까요?"
- "navbar에 로그인 상태를 표시할까요?"
- "하이프 보내기 UI는 어떤 형태?"

**판단 기준:**
1. **기획서/와이어프레임에 언급이 있으면** → 그에 맞게
2. **사용자 경험 관점에서 자연스러운 쪽** → 보통 "(권장)" 옵션
3. **둘 다 합리적이면** → "(권장)" 선택
4. **디자인 시스템이나 기존 패턴이 있으면** → 일관성 유지

### 2.3 기술 선택

**실제 예시:**
- "로그인 방식은 어떤 걸 지원할까요? Google만 / Google+이메일 / Google+패스워드리스"

**판단 기준:**
1. **기존 코드에 이미 구현된 것이 있으면** → 그것 기반으로
2. **프로젝트 스펙에 명시되어 있으면** → 따름
3. **없으면** → 가장 단순한 옵션 (복잡도 최소화)

### 2.4 확인/진행 질문

**실제 예시:**
- "스펙 1 디자인을 이대로 확정하고 문서로 작성할까요?"

**판단 기준:**
1. **"확정하고 진행"** vs **"수정할 부분 있음"** → 특별한 이슈가 없으면 **진행**
2. 프로젝트 문서에서 명시적으로 다른 방향을 지시하면 → "수정"

### 2.5 정보 입력 요청

**실제 예시:**
- "Firebase 프로젝트 정보를 알려주세요"
- "환경변수 파일을 어떤 형태로 만들까요?"

**판단 기준:**
1. **시크릿/크레덴셜 요청** → 무조건 ESCALATE (사람이 직접 입력)
2. **템플릿 vs 실제 값** → 항상 "템플릿" 선택 (안전)
3. **설정 형식 선택** → 프로젝트 기존 패턴 따름

### 2.6 범용 규칙 (모든 AskUserQuestion에 적용)

```
우선순위:
1. golden-rules 블랙리스트에 해당하면 → ESCALATE
2. 프로젝트 문서에 명시적 답이 있으면 → 그것
3. (권장) 표시 옵션이 있으면 → 그것
4. 위 조건 모두 없으면 → 가장 보수적/단순한 옵션
5. 판단 불가능하면 → ESCALATE
```

## 3. 자유 텍스트 입력 (Free Text) — 매우 드묾

현재 분석된 4개 세션에서 자유 텍스트 입력 대기 상태는 관찰되지 않음.
발생 시:

1. `docs/exec-plans/active/` 에 활성 실행 계획이 있으면 → 다음 단계 지시
2. 최근 대화 맥락에서 Cowork가 뭘 기다리는지 파악 → 적절한 지시
3. 판단 불가능하면 → ESCALATE

---

## 부록: ESCALATE 응답 형식

CLI 에이전트가 "ESCALATE"를 응답하면, 오케스트레이터는 해당 질문을 건너뛰고
사람에게 알림을 보낸다. CLI는 다른 텍스트 없이 정확히 `ESCALATE`만 리턴해야 한다.
